> Operative direction for the project. Generated by an adversarial planning workflow (5 work-streams -> value-first positioning -> adversarial critique -> synthesis), then extended with a concrete file-cleanup list and the standing review gate. **Follow this as the roadmap.**

# Loqi Master Plan — "jq for the text only an LLM can parse"

## 1. North star + one-job identity

**North star:** A single static binary that turns messy text into schema-validated NDJSON in one pipe stage — `cat tickets.txt | loqi extract.lq > tickets.ndjson` — where each row matches the schema or the run stops with a caret naming the field that broke.

**One job:** Loqi reads free-form text on stdin, runs an LLM extraction per record in parallel, and emits structured rows the next pipe stage can trust — validated-or-fails, no runtime, no SDK, no venv. It is still a real language; the script *is* the filter.

## 2. Milestones

> **Reframing vs. the work-streams as written.** The critique is verified against `src/loqi.c` and is correct on the load-bearing points, so this plan adopts its corrections:
> - The README hero's `ai_json.validated(row, schema)` and `lines(stdin())`-as-lazy-iterator **do not exist**. But `ai_json(prompt, schema)` (line 4700) **already** does the model call + schema-check + one retry + raise-with-field-detail. **That is the validate-or-fail primitive. Use it. Do not invent a new API.**
> - `lines()` (line 5684) already splits a string and handles CRLF + trailing newline. The only missing input piece is a zero-arg **`stdin()` slurp** native (S-effort), **not** NF2's lazy `ObjStream` + `OP_FOR_NEXT` surgery (which touches the absolute-slot footgun CLAUDE.md flags). NF2 is **cut from MVP**.
> - `ai_all` returns a whole batch and longjmps on the first bad row, so **"stream rows as they validate" is architecturally false today.** Drop the word "streaming" from all copy. The MVP hero loops `ai_json` per line (honest, sequential, validated). `ai_all` parallelism is a fast-follow once a per-row emit-then-continue loop exists.
> - **Linux builds belong in M0**, not later — the entire product promise is "one static binary, no runtime." A macOS-only binary cannot deliver the wedge to the container/CI/server audience that lives in pipes. De-Appling the build is one flag and is the named gate.

## 4. Final README hero + GitHub About (ready to paste)

**README hero** — note this version uses **only the verified API** (`stdin()` slurp from M0-3, `lines()` line 5684, `ai_json(prompt, schema)` line 4700, `json.stringify`), so it runs verbatim once M0 lands. No `ai_all`, no `.validated()`, no "streaming":

```markdown
# Loqi

**jq for the text only an LLM can parse.** Pipe in messy text, get back schema-validated
structured rows (NDJSON) your next pipe stage can trust — or a hard failure that names the
field that didn't validate.

One static binary. No Python, no venv, no SDK, no runtime to install. The script *is* the filter.

```sh
cat tickets.txt | loqi extract.lq > tickets.ndjson
```

```loqi
# extract.lq — one record per input line, validated against a schema,
# emitted as NDJSON. A row that fails the schema stops the run with a caret.
let schema = {
  type: "object",
  required: ["id", "severity", "summary"],
  fields: {
    id:       { type: "int" },
    severity: { type: "string", enum: ["low", "medium", "high", "critical"] },
    summary:  { type: "string", pattern: ".{3,}" },
  },
}

let opts = { model: "claude-sonnet-4-5", temperature: 0 }   # pinned: same input -> same row

for line in lines(stdin()) {
  let row = ai_json("Extract id, severity and summary from this ticket:\n{line}", schema, opts)
  print(json.stringify(row))   # one NDJSON line
}
```

```
$ cat tickets.txt | loqi extract.lq > tickets.ndjson
{"id":4821,"severity":"high","summary":"Checkout 500s on Safari after coupon applied"}
{"id":4822,"severity":"low","summary":"Typo in the footer copyright year"}

$ cat tickets.txt | loqi extract.lq > out.ndjson
extract.lq:18:13: ai_json(): output did not match the schema after a retry
  severity: "blocker" is not one of the allowed enum values
            ^
exit status 70
```

**Validated-or-fails:** every row matches the schema (with a retry), or the run stops with a
caret on the field that broke. The next pipe stage never eats half-parsed garbage.

**Why not Python + an SDK?** In a pipe you pay the venv/pip/import tax, then write your own
client, JSON parsing, schema check, retries and parallelism. Loqi bakes that into one binary
you drop on any box: typed, validated, retry-on-429.

**Why not jq?** jq needs structured input. Loqi *produces* it — from the free-form text jq,
grep and awk can read but can't reason about.

## Install
macOS (arm64), Linux (x86_64, arm64) — download the static binary from
[Releases](…) or `curl -fsSL https://<pages>/install.sh | sh`.
Runtime needs: `curl` on PATH and `ANTHROPIC_API_KEY` in the env. That's it.
```

## 7. Continuous code review & code-quality review (standing gate)

Every feature and every milestone passes through the same loop that already caught real HIGH bugs this session (the `http.serve` try/catch bypass, CRLF header injection, the line-continuation stack overflow, the caret line/col mismatch):

**Per change:** `build -> adversarial multi-agent review -> fix confirmed findings -> verify -> commit.**

1. **Code review (correctness + memory-safety):** an adversarial multi-agent pass over the *changed paths only* (not a full re-review), each finding empirically verified — reproduce under ASan/UBSan, `leaks --atExit`, and the curl-stub for AI paths — before it is accepted. Default a claimed bug to "refuted" unless the repro holds.
2. **Code-quality review:** single-file stays cleanly sectioned and readable; no dead code; naming/idiom consistent with the surrounding code; docs (README/STDLIB/LANGUAGE/CHANGELOG) kept in sync with the change; **no em dashes anywhere**; honest claims only (every claim backed by code + a test).
3. **Verify (definition of done, before commit, in a separate step):** full `test.sh` green **plus the new tests**, on macOS arm64 AND the actual musl Linux artifacts; ASan/UBSan clean; `leaks --atExit -> 0` (incl. a forced validation-failure run, the malloc-before-longjmp shape); LSan clean on Linux.
4. **Release discipline (CLAUDE.md):** never bump/tag on a red gate; one cumulative release per milestone, not a tag per task; annotated tags, never force-pushed; README is part of the release surface.

This gate is non-negotiable and applies to M0..M3.
